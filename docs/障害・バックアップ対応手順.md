# KenSapo 障害・バックアップ対応手順

## 1. 問い合わせ時に確認すること

利用者から次を聞き取ります。

- 発生した日時
- 使用端末（iPhone、Android、PC）
- 画面名
- 押したボタン
- 表示された文言
- 全員で発生しているか、1人だけか
- スクリーンショット

## 2. 障害の切り分け

### 画面が開かない

1. https://kensapo.vercel.app が開くか確認します。
2. VercelのDeployment Statusを確認します。
3. `/api/health/db` が `{"ok":true}` を返すか確認します。
4. Supabase StatusとVercel Statusを確認します。

### ログインできない

1. メールアドレスの入力ミスを確認します。
2. Supabase Authenticationでユーザーの存在を確認します。
3. メール確認設定を確認します。
4. 必要ならパスワード再設定を行います。

### データが見えない

1. 正しい会社アカウントか確認します。
2. `profiles.company_id` を確認します。
3. RLS設定と対象データの `company_id` を確認します。

### 写真が保存できない

1. 通信状態とファイルサイズを確認します。
2. Supabase Storageの容量とポリシーを確認します。
3. `site-photos` バケットの存在を確認します。

### 通知が届かない

1. `push_subscriptions` テーブルの存在を確認します。
2. 受信端末で通知がオンか確認します。
3. 端末OSの通知設定を確認します。
4. iPhoneではホーム画面から起動しているか確認します。
5. VercelのVAPID環境変数を確認します。

## 3. 復旧時の原則

- 本番データを直接削除しません。
- 修正前に対象レコードを控えます。
- Gitのmainブランチへ直接危険な変更を入れません。
- 本番反映前に `npm run build` を成功させます。
- 修正後は問題の操作を本番で再確認します。

## 4. バックアップ

### 定期確認

月1回、次を確認します。

- Supabaseの利用プラン
- Database Backupsの有効状態
- Storage使用量
- Database使用量
- Vercelの最新正常Deployment
- GitHubへ最新コードがpushされていること

### 手動退避

重要な納品前・大規模変更前には、Supabase Dashboardから主要テーブルをCSV出力するか、利用プランに応じたバックアップ機能を使用します。

対象:

- companies
- profiles
- projects
- workers
- schedules
- site_photos
- daily_reports
- project_progress_items
- push_subscriptions

写真本体はStorageに保存されているため、データベースのCSVだけでは写真を復元できません。

## 5. 連絡優先度

- **緊急**: 全員ログイン不可、全データ閲覧不可
- **高**: 予定・写真・日報を保存できない
- **中**: 一部端末で通知が届かない、表示崩れ
- **低**: 文言修正、使い方質問、改善要望

## 6. 障害記録

障害ごとに次を残します。

- 発生日
- 影響会社・利用者
- 症状
- 原因
- 対応内容
- 復旧時刻
- 再発防止策
